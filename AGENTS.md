# Vera Development Constraints

These rules apply to the entire repository.

1. Read [PRODUCT.md](./PRODUCT.md) before starting any task. Treat it as the sole authoritative product context.
2. Before modifying UI, obtain the corresponding Mike page or saved Mike screenshot as visual truth. Record a same-size Mike/Vera comparison for acceptance.
3. Reuse existing Mike pages, routes, data contracts, and components by default. Vera-specific UI is a minimal legal capability increment, not a parallel design system.
4. Before adding a table, state, page, panel, or general framework, demonstrate that the existing task, artifact, document, citation, review, or Matter objects cannot carry the requirement.
5. Every feature must shorten completion time, improve work-product quality, improve source verification, or control a consequential final action. If it satisfies none, do not implement it.
6. The normal path shows work and outputs. Security, local-storage, permission, hash, and audit detail appears only for exceptions, export/egress, or explicit expansion.
7. Do not change APIs, permissions, security boundaries, authentication, export gates, or legal disclaimers merely to support a UI adjustment.
8. Preserve unknown and unrelated untracked files. Do not clean, move, stage, or delete them.
9. UI acceptance must cover same-size Mike/Vera screenshots, keyboard operation and visible focus, long Chinese text, 125% and 150% zoom, and responsive behavior. Keep the primary workflow recognizable as Mike.
